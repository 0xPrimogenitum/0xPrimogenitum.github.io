0xPrimogenitum

I care more about understanding systems than using them.

Philosophy

Most software is designed to hide complexity. That's useful for productivity, but it creates a dependency on assumptions you didn't make. When something breaks or behaves unexpectedly, you're stuck guessing. I prefer to know what's actually happening under the surface.

Rebuilding things from scratch isn't about reinventing the wheel. It's about understanding why the wheel works. You learn what decisions were made, what tradeoffs exist, and where the constraints actually are. That knowledge makes you less reliant on documentation and better at diagnosing problems.

How I Learn

Breaking systems is the fastest way to understand their boundaries. When you push something until it fails, you learn what it was designed to handle and what it wasn't. The failure modes tell you more than the documentation ever will.

Observation matters more than theory. You can read about how a protocol works, but watching the actual packets move shows you what's really happening. Behavior is ground truth. Everything else is interpretation.

Rebuilding forces you to make the same decisions the original authors did. You hit the same problems, find the same edge cases, and understand why certain patterns exist. It's slower than using a library, but the knowledge stays with you.

What I Enjoy Building

Internals are more interesting than interfaces. The public API is what someone decided you should see. The implementation is where the actual work happens. Game engines, file formats, network protocolsâ€”the internal structure is where you find the real constraints and clever solutions.

Tooling that exposes what's hidden is worth building. Debuggers, disassemblers, memory viewers, protocol analyzers. Anything that makes invisible state visible or lets you interact with a system at a lower level than it was designed for.

Low-level systems programming keeps you close to what the machine is actually doing. There's less abstraction between your code and the hardware. You see the cost of every operation. The feedback loop is tighter and the decisions feel more concrete.

What I Avoid

Overengineering happens when you're solving imaginary problems. You build for scale you'll never hit or flexibility you'll never need. The complexity becomes its own maintenance burden. It's better to solve the problem in front of you and refactor when you actually know what's needed.

Blind dependencies mean you're trusting code you haven't read. That's sometimes necessary, but it shouldn't be the default. When something goes wrong, you need to know if the problem is yours or theirs. If you can't read the source and understand what it's doing, you're just hoping it works.

The "just works" mentality treats tools like magic. When things work, you don't learn anything. When they break, you don't know where to start. Understanding the mechanism makes you self-sufficient. You can fix it, modify it, or replace it if needed.
